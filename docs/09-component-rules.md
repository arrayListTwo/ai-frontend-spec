# 09 组件开发规则（Component Development Rules）

## 目标

本规范定义 AI 在开发、修改和维护组件时应遵循的统一规则。

`07-coding-rules` 约束的是**组件代码如何编写**。

本规范约束的是**组件应该如何设计、拆分、复用和演进**。

AI 在开发组件之前，应首先判断是否真的需要创建新组件，而不是默认新增组件。

一句话：

> 组件负责封装能力，而不是封装整个业务。

---

# 组件职责

组件是页面能力的组成单元。

组件负责：

* UI 展示
* 用户交互
* 局部状态
* 对外暴露能力
* 提供复用能力

组件不负责：

* 页面组织
* 全局状态
* 页面权限
* 页面生命周期
* 复杂业务流程
* 页面之间通信

组件应尽量保持独立，可在不同页面中复用。

---

# 组件分类

建议将组件分为三类。

## 一、基础组件（Base Components）

特点：

* 无业务属性
* 高复用
* 与具体业务无关

例如：

* SvgIcon
* Pagination
* ImagePreview
* Upload
* DictTag
* Empty
* SearchBar

目录：

```text
src/components
```

---

## 二、业务组件（Business Components）

特点：

* 与某一业务相关
* 通常只服务于一个模块
* 可以被同模块多个页面复用

例如：

```text
UserForm

RoleTree

DeptSelector

AssignRoleDialog
```

目录：

```text
views/system/user/components
```

不要放入：

```text
src/components
```

---

## 三、页面容器组件（Page Components）

特点：

负责组织页面中的局部区域。

例如：

* SearchPanel
* Toolbar
* StatisticsCard
* ApprovalTimeline

通常：

仅当前页面使用。

---

# 创建组件之前

AI 必须依次检查：

1. 是否已有相同组件
2. 是否已有类似组件
3. 是否可以扩展已有组件
4. 是否真的需要新增组件

不要：

为了几十行代码：

新增一个组件。

---

# 单一职责

一个组件：

只完成一项能力。

例如：

正确：

```text
UserForm
```

负责：

编辑用户。

错误：

```text
UserComponent
```

同时负责：

* 查询
* 表格
* 编辑
* 删除
* 导入
* 导出
* 权限管理

---

# 输入与输出

统一采用：

Props 输入。

Emits 输出。

推荐数据流：

```text
Page
 │
 ▼
Props
 │
 ▼
Component
 │
 ▼
Emit
 │
 ▼
Page
```

不要：

组件直接修改页面状态。

---

# Props 设计

Props 应满足：

* 数量适中
* 语义清晰
* 默认值合理
* 类型明确
* 尽量保持只读

不要：

通过大量布尔值控制组件行为。

例如：

```text
editable
readonly
preview
disabled
simple
advanced
mini
popup
dialog
drawer
```

如果布尔参数越来越多，应考虑重新设计组件。

---

# Emit 设计

事件名称应体现业务语义。

推荐：

* submit
* cancel
* success
* close
* confirm
* change
* refresh

不要：

```text
ok

done

event

callback
```

---

# 插槽设计

当组件存在可扩展区域时，应优先考虑插槽。

常用插槽：

* default
* header
* footer
* empty
* toolbar

不要：

为了少量 UI 差异复制整个组件。

---

# 组件状态

组件仅维护局部状态。

例如：

* 当前 Tab
* 当前步骤
* 输入值
* 展开状态
* 局部 Loading

不要维护：

* 用户信息
* Token
* 权限
* 系统配置

这些属于 Store。

---

# API 调用

默认：

组件不直接调用业务 API。

推荐：

```text
页面

↓

请求 API

↓

组件展示
```

特殊情况：

允许：

* 上传组件
* 远程搜索
* 懒加载树
* 图片预览
* 独立业务组件

但应保持输入输出明确。

---

# Store 使用

默认：

组件不直接依赖 Store。

优先：

由页面传入数据。

只有真正属于全局状态时：

组件才读取 Store。

不要：

为了减少 Props：

直接读取 Store。

---

# Router 使用

默认：

组件不负责页面导航。

页面：

负责：

Router。

组件：

负责：

事件通知。

---

# 生命周期

组件生命周期：

仅负责：

初始化自己。

不要：

组件初始化：

加载整个页面。

不要：

组件初始化：

修改全局状态。

---

# defineExpose

只暴露：

必要方法。

例如：

* open()
* close()
* reset()
* validate()
* focus()

不要：

暴露：

全部变量。

---

# 组件拆分

建议拆分条件：

* 超过 300 行
* 多个职责
* 多个独立区域
* 可复用
* 可独立维护

不要：

为了拆分而拆分。

---

# 组合优于继承

推荐：

多个小组件：

组合。

不要：

一个超级组件：

通过：

```text
type

mode

scene
```

切换：

十几种模式。

---

# 样式

组件：

维护：

自己的样式。

不要：

依赖：

页面样式。

不要：

修改：

其它组件样式。

---

# 测试性

组件应尽量满足：

* 独立渲染
* 独立输入
* 独立输出
* 无隐藏副作用

避免：

组件依赖大量全局环境。

---

# 可维护性

组件应具备：

* 清晰命名
* 清晰职责
* 易定位问题
* 易复用
* 易扩展
* 易测试

---

# 组件演进原则

已有组件：

优先：

扩展。

不要：

复制一份：

修改。

例如：

不要：

```text
UserForm.vue

UserForm2.vue

UserFormNew.vue
```

应：

统一维护。

---

# AI 组件开发流程

AI 在开发组件时，应遵循以下顺序：

1. 确认是否已有同类组件
2. 判断是否需要新建组件
3. 确定组件职责
4. 设计 Props
5. 设计 Emits
6. 判断是否需要 Slot
7. 判断是否需要暴露方法
8. 实现局部状态
9. 编写组件样式
10. 检查可复用性
11. 检查与项目风格一致性

不得直接开始编码。

---

# AI 自动生成要求

AI 在生成组件时必须遵守：

* 优先复用已有组件
* 单一职责
* Props 输入
* Emits 输出
* 默认不调用业务 API
* 默认不依赖 Store
* 默认不负责 Router
* 保持组件独立
* 保持组件易复用
* 保持组件易测试
* 保持组件易维护
* 保持与项目整体风格一致

---

# AI 禁止行为

AI 不得生成以下组件：

* 一个组件承担多个业务
* 组件直接修改页面状态
* 组件承担页面导航
* 组件维护全局状态
* 组件大量依赖 Store
* 组件与页面强耦合
* 一个组件拥有大量模式切换
* 组件无限扩展而不拆分
* 重复创建已有组件
* 复制已有组件形成多个版本
* 与项目现有组件规范不一致

---

# Checklist

AI 完成组件开发后，应确认以下内容：

* 已确认不存在可复用组件
* 组件职责单一
* 分类正确（基础组件、业务组件或页面容器组件）
* Props 设计合理且语义清晰
* Emits 命名符合业务语义
* 必要时提供 Slot，而非复制组件
* 默认未直接调用业务 API
* 默认未依赖全局 Store
* 默认未负责页面导航
* `defineExpose` 仅暴露必要能力
* 样式独立，不污染外部
* 组件具备良好的复用性、可维护性和可测试性
* 未产生重复组件或多个功能相同的组件版本
* 与项目整体组件设计规范保持一致
