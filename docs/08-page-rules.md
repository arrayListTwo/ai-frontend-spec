# 08 页面开发规则（Page Development Rules）

## 目标

本规范定义 AI 在开发、修改和维护页面时应遵循的统一规则。

`07-coding-rules` 约束的是**代码如何编写**。

本规范约束的是**页面应该如何组织**。

AI 在生成任何页面之前，应优先理解页面职责，再按照统一的页面结构进行开发，而不是根据个人偏好自由发挥。

一句话：

> 页面负责组织业务，而不是承载所有业务。

---

# 页面职责

页面（View）是业务的组织者。

页面负责：

* 页面布局
* 页面生命周期
* 页面权限控制
* 页面状态管理
* 页面数据加载
* 调用 API
* 调用 Store
* 调用 Composable
* 调用组件
* 页面之间导航

页面不负责：

* HTTP 请求实现
* 公共算法
* 公共组件实现
* 工具函数实现
* 全局状态维护
* 通用业务能力封装

---

# 页面目录

推荐目录结构：

```text
views
└── system
    └── user
        ├── index.vue
        ├── components
        │   ├── EditDialog.vue
        │   ├── ImportDialog.vue
        │   └── AssignRoleDialog.vue
        ├── hooks
        │   └── useUserTable.js
        └── constants.js
```

推荐原则：

* 一个菜单对应一个目录
* 一个目录对应一个业务
* 页面入口统一为 `index.vue`

不要：

```text
User.vue
Main.vue
List.vue
Home.vue
```

作为业务入口。

---

# 页面组成

一个标准页面建议包含以下部分：

1. 查询区域（Search）
2. 工具栏（Toolbar）
3. 数据表格（Table）
4. 分页（Pagination）
5. 弹窗（Dialog）
6. 抽屉（Drawer，可选）
7. 业务组件（Components）

保持所有后台页面结构一致。

---

# 页面数据流

统一数据流如下：

```text
用户操作
    │
    ▼
页面事件
    │
    ▼
调用 API / Store
    │
    ▼
更新页面状态
    │
    ▼
刷新页面
```

不要：

```text
组件
    │
    ▼
请求 API
    │
    ▼
修改 Store
    │
    ▼
刷新页面
```

数据流应保持单向、清晰、可追踪。

---

# 页面初始化

推荐流程：

1. 初始化响应式状态
2. 获取必要参数
3. 初始化权限
4. 请求页面数据
5. 初始化完成

不要：

在 `onMounted()` 中执行大量无关逻辑。

复杂初始化建议拆分为多个独立方法。

---

# 查询区域

查询区域职责：

* 输入查询条件
* 查询
* 重置
* 展开/收起（如项目支持）

不要：

在查询区域：

* 请求多个接口
* 写复杂业务判断
* 写权限逻辑

---

# 数据加载

页面统一提供：

* `loading`
* `list`
* `total`

加载流程建议：

1. 设置 `loading = true`
2. 调用 API
3. 更新数据
4. 处理异常
5. `loading = false`

不要：

多个地方分别维护不同的主 Loading 状态。

---

# 页面状态

页面状态包括：

* 查询条件
* 表单数据
* 当前分页
* Dialog 状态
* 当前选中数据
* Loading
* 当前 Tab

不要：

把这些状态放入 Pinia。

---

# 页面事件

页面事件建议按照职责划分：

数据：

* 获取列表
* 获取详情

查询：

* 查询
* 重置

操作：

* 新增
* 修改
* 删除
* 导入
* 导出

弹窗：

* 打开
* 关闭
* 提交

保持分类清晰。

---

# 页面组件拆分

满足以下任一条件，应拆分组件：

* 超过 300 行
* 存在多个 Dialog
* 存在复杂表单
* 存在复杂表格
* 同一模块可以独立维护
* 有复用价值

不要：

为了拆分而拆分。

---

# 页面之间通信

推荐：

* Router
* Pinia（共享状态）
* API

不要：

* 全局变量
* Window
* LocalStorage（一次性通信）
* 直接访问其它页面实例

---

# 页面权限

统一使用项目权限体系。

例如：

* 按钮权限
* 菜单权限
* 数据权限

不要：

在页面中自行维护权限判断。

例如：

不要到处写：

```text
if (user.role === 'admin')
```

统一复用权限机制。

---

# 页面异常处理

页面负责：

* 请求异常提示
* 操作失败提示
* 空状态展示
* 加载状态恢复

不要：

吞掉异常。

不要：

空 `catch`。

---

# 页面刷新

数据修改成功后：

优先：

局部刷新。

例如：

* 刷新列表
* 更新当前记录

避免：

整个页面重新加载。

除非业务确有需要。

---

# 页面导航

统一通过项目路由。

不要：

使用：

* `window.location`
* `location.href`

进行站内跳转。

---

# 页面缓存

是否缓存页面：

由项目路由配置决定。

页面不要：

自行实现缓存逻辑。

---

# 页面响应式

后台管理系统：

默认优先桌面端体验。

如需移动端支持，应遵循项目统一响应式方案。

不要：

每个页面实现不同的响应式规则。

---

# 页面性能

避免：

* 重复请求接口
* 重复计算数据
* 重复创建对象
* 无意义 Watch

优先：

* computed
* 缓存已有结果
* 复用已有数据

---

# 页面可维护性

页面代码应具备：

* 清晰目录
* 清晰命名
* 清晰职责
* 易于定位问题
* 易于 AI 修改
* 易于团队维护

不要：

所有逻辑都堆积在 `index.vue`。

---

# AI 页面开发流程

AI 在开发页面时应遵循以下顺序：

1. 理解页面所属业务模块
2. 复用已有页面模板
3. 检查是否已有可复用组件
4. 检查是否已有 API
5. 检查是否已有 Store
6. 设计页面状态
7. 实现页面布局
8. 接入数据请求
9. 接入交互逻辑
10. 检查命名一致性
11. 检查异常处理
12. 检查是否符合项目规范

不得跳过前面的分析步骤直接生成代码。

---

# AI 自动生成要求

AI 在生成页面时必须遵守：

* 优先复用已有页面结构
* 保持页面职责单一
* 保持统一目录结构
* 保持统一命名规范
* 不直接实现 HTTP 请求
* 不重复实现公共逻辑
* 不重复实现公共组件
* 优先拆分复杂模块
* 保持页面结构一致
* 保持数据流单向
* 保持与项目整体风格一致

---

# AI 禁止行为

AI 不得生成以下页面：

* 页面直接调用 axios
* 页面直接调用 fetch
* 页面直接维护全局状态
* 页面包含大量公共工具函数
* 页面承担多个业务模块
* 页面与组件高度耦合
* 页面依赖其它页面内部实现
* 页面包含重复代码
* 页面自行实现权限体系
* 页面直接操作浏览器地址完成站内跳转
* 页面无限堆积逻辑而不拆分
* 与项目现有页面结构不一致

---

# Checklist

AI 完成页面开发后，应确认以下内容：

* 页面目录结构符合项目规范
* 页面职责清晰且单一
* 页面数据流清晰、单向
* 页面未直接发送 HTTP 请求
* 页面优先复用已有组件、API 与 Store
* 页面状态仅包含局部业务状态
* 页面事件分类合理
* 查询、表格、分页、弹窗结构清晰
* 权限控制复用项目已有机制
* 异常处理完整
* 数据刷新策略合理
* 页面未包含重复逻辑
* 页面易于扩展、维护与测试
* 与项目整体开发规范保持一致
