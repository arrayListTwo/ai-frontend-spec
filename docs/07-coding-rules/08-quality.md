# 08 代码质量规范（Code Quality Rules）

## 目标

本规范用于约束 AI 在编写、修改和重构代码时的质量要求。

目标包括：

* 保持代码一致性
* 提高代码可读性
* 提高代码可维护性
* 降低 Bug 产生概率
* 降低 AI 重复生成低质量代码
* 保证新增代码能够自然融入现有项目

本规范适用于：

* Vue 页面
* Vue 组件
* Pinia Store
* API
* Hooks（Composable）
* Utils
* 指令
* 插件
* TypeScript（如项目使用）
* JavaScript

一句话：

> AI 编写的代码，应像资深开发者维护多年的项目代码，而不是一次性生成的代码。

---

# 核心原则

AI 必须遵循以下原则：

1. 优先修改已有代码，而不是重新实现
2. 保持项目已有编码风格
3. 不为了"更高级"而重构
4. 不增加无意义抽象
5. 不增加无意义依赖
6. 保持代码简单、稳定、可维护
7. 每一行新增代码都应有明确价值

---

# 一致性优先

当项目已有实现方式时：

AI 必须：

优先保持一致。

例如：

已有：

```text
async/await
```

不要：

新写：

```text
Promise.then()
```

已有：

```text
Composition API
```

不要：

新增：

```text
Options API
```

已有：

Pinia。

不要：

新增：

Vuex。

---

# 最小修改原则

修改 Bug 时：

优先：

最小改动。

不要：

为了修改一行代码：

重写整个模块。

例如：

正确：

```text
修改一个判断条件
```

错误：

```text
整个页面全部重构
```

---

# 不重复实现

发现已有：

* 工具函数
* Hook
* API
* Store
* 组件

必须：

优先复用。

不要：

重新写一份。

---

# 单一职责

一个：

函数：

完成：

一件事情。

不要：

一个函数：

同时：

* 查询
* 校验
* 保存
* 删除
* 导出

全部完成。

---

# 函数长度

建议：

30 行以内。

超过：

建议：

拆分。

复杂流程：

拆成：

多个：

私有函数。

不要：

一个函数：

200 行。

---

# 嵌套层级

建议：

最多：

三层。

例如：

```text
if

↓

for

↓

if
```

超过：

考虑：

提前返回。

抽离函数。

优化条件。

---

# 提前返回

推荐：

使用：

Early Return。

例如：

```text
参数非法

↓

直接返回
```

避免：

大量：

```text
if

else

else

else
```

不断嵌套。

---

# 魔法值

不要：

直接写：

```text
0

1

2

99

1000
```

没有业务意义。

建议：

提取：

常量。

例如：

* 默认分页大小
* 最大上传数量
* 状态码
* 超时时间

---

# 字符串

不要：

重复：

硬编码。

例如：

```text
success

error

admin

enable
```

建议：

统一管理：

枚举或常量。

---

# 条件判断

保持：

简单。

不要：

一个：

if

包含：

多个：

复杂表达式。

建议：

拆分：

命名变量。

提高可读性。

---

# Boolean 命名

推荐：

```text
isLoading

isAdmin

hasPermission

canEdit

shouldRefresh
```

不要：

```text
flag

status

ok

type
```

表达真假。

---

# Null 判断

统一：

判空。

不要：

有时：

```text
== null
```

有时：

```text
=== undefined
```

有时：

```text
=== ''
```

混用。

保持项目一致。

---

# Optional Chaining

优先：

使用：

```text
?.
```

避免：

连续：

```text
&&
&&
&&
```

判断。

---

# Nullish Coalescing

优先：

使用：

```text
??
```

不要：

大量：

```text
||
```

导致：

0

false

''

被误判。

---

# 异常处理

所有：

异步逻辑：

必须：

处理异常。

不要：

忽略：

Promise。

不要：

空：

catch。

例如：

不要：

```text
catch (e) {}
```

---

# 日志

开发日志：

允许：

```text
console.log()
```

提交前：

应删除。

保留：

必要：

```text
console.error()

console.warn()
```

并确保符合项目规范。

不要：

页面：

充满：

调试日志。

---

# 注释

保留：

有价值：

注释。

例如：

* 特殊业务
* 临时兼容
* 性能优化
* 第三方限制

不要：

解释：

代码本身。

例如：

```text
点击按钮

获取数据

循环数组
```

---

# TODO

允许：

保留：

TODO。

必须：

说明：

原因。

例如：

```text
TODO：等待后端支持批量接口后优化
```

不要：

只有：

```text
TODO
```

---

# Import

保持：

排序统一。

建议：

顺序：

1. Vue
2. 第三方库
3. API
4. Store
5. Hooks
6. Utils
7. Components
8. Styles

不要：

随机顺序。

---

# 未使用代码

禁止：

新增：

未使用：

* import
* ref
* computed
* 方法
* 常量
* 变量

保持：

代码干净。

---

# 命名

所有命名：

必须：

具有业务语义。

不要：

```text
a

b

tmp

obj

list1

newData
```

---

# 重复代码

出现：

三次以上：

建议：

抽离。

不要：

复制。

例如：

重复：

Message。

重复：

格式化。

重复：

转换。

---

# 性能

不要：

提前：

做：

性能优化。

仅：

真正存在：

性能问题时：

再优化。

不要：

为了性能：

牺牲：

可读性。

---

# 响应式

避免：

无意义：

响应式。

例如：

永远不会变化：

的数据。

不要：

放：

ref。

---

# 深拷贝

不要：

无意义：

JSON：

深拷贝。

优先：

项目已有工具。

或：

浏览器原生：

`structuredClone()`（兼容性允许时）。

---

# 工具函数

工具函数：

保持：

纯函数。

不要：

依赖：

页面。

不要：

依赖：

Store。

不要：

依赖：

DOM。

---

# Hooks（Composable）

Composable：

负责：

逻辑复用。

不要：

负责：

页面。

不要：

包含：

大量 UI。

不要：

成为：

万能工具。

---

# 可测试性

新增代码：

应尽量：

容易测试。

避免：

大量：

隐藏副作用。

避免：

修改：

全局变量。

---

# 向后兼容

修改已有功能：

不得：

破坏：

已有：

接口。

已有：

Props。

已有：

事件。

已有：

返回值。

除非：

需求明确要求。

---

# 重构原则

允许：

重构。

前提：

* 不改变行为
* 不改变接口
* 不增加复杂度
* 可明显提高可读性

不要：

为了：

"代码优雅"

重写整个模块。

---

# AI 自动生成要求

AI 在生成代码时必须遵守：

* 优先复用已有实现
* 保持最小修改原则
* 保持项目编码风格一致
* 保持函数职责单一
* 控制函数长度与嵌套层级
* 使用语义化命名
* 不新增未使用代码
* 保持导入顺序统一
* 保持异常处理完整
* 保持代码易读、易维护
* 不引入无意义抽象
* 不引入无意义依赖
* 不改变已有行为

---

# AI 禁止行为

AI 不得生成以下代码：

* 大规模无必要重构
* 重复实现已有能力
* 一个函数承担多个职责
* 深层嵌套逻辑
* 大量魔法值
* 大量硬编码字符串
* 空 `catch`
* 大量调试日志
* 无意义注释
* 未使用的变量、方法或导入
* 无语义命名
* 为优化而优化
* 修改已有公共接口行为
* 引入新的框架或状态管理方案
* 与项目既有编码风格不一致

---

# Checklist

AI 完成编码后，应确认以下内容：

* 优先复用了项目已有实现
* 修改范围符合最小修改原则
* 保持与项目编码风格一致
* 函数职责单一，长度合理
* 逻辑层级清晰，避免过度嵌套
* 使用语义化命名
* 魔法值已提取为常量（适用时）
* 条件判断清晰易读
* 异步逻辑具备必要异常处理
* 无未使用的变量、方法或导入
* 导入顺序统一
* 注释具有实际价值
* 工具函数保持纯函数特性
* Composable 职责明确
* 未破坏已有接口、Props 或事件
* 未引入无意义抽象或额外依赖
* 新增代码易于阅读、维护与后续扩展
