# 13 AI 自检规范（AI Self Review）

## 目标

本规范定义 AI 在完成任何代码生成、修改、重构或修复任务后，必须执行的自检流程。

AI 不应在代码生成完成后立即结束任务，而应像经验丰富的开发者一样，对结果进行全面检查，确保代码质量、架构一致性和项目规范均符合要求。

本规范适用于：

* 新功能开发
* Bug 修复
* 页面修改
* 组件开发
* API 开发
* Store 修改
* 样式调整
* 重构优化

一句话：

> AI 不仅要会写代码，更要会检查自己写的代码。

---

# 自检原则

AI 必须遵循以下原则：

1. 每次完成任务都必须执行自检
2. 自检优先于继续开发
3. 优先发现问题，而不是证明自己正确
4. 发现问题应优先修复
5. 修复后重新执行完整自检

不得因为代码已经能够运行而跳过检查。

---

# 第一阶段：需求检查

确认是否真正完成了用户需求。

检查内容：

* 是否遗漏任何需求
* 是否误解需求
* 是否新增了用户没有要求的功能
* 是否改变了已有业务行为
* 是否影响已有功能

目标：

确保实现的是用户需要的，而不是 AI 自己认为合理的方案。

---

# 第二阶段：架构检查

确认是否符合项目架构。

检查内容：

* 是否放在正确目录
* 是否遵循项目目录规范
* 是否符合模块划分
* 是否符合职责边界
* 是否新增了不必要的模块
* 是否新增了重复能力

目标：

保证新增代码能够自然融入项目。

---

# 第三阶段：复用检查

确认是否存在重复实现。

检查内容：

* 是否已有类似组件
* 是否已有类似 API
* 是否已有类似 Store
* 是否已有类似工具函数
* 是否已有类似 Hook（Composable）
* 是否已有公共样式

如果已有：

优先复用。

不要重复实现。

---

# 第四阶段：命名检查

检查所有新增命名。

包括：

* 文件
* 文件夹
* 方法
* 变量
* Props
* Emits
* Store
* API
* Class

要求：

* 具有业务语义
* 保持项目一致
* 无缩写歧义
* 无临时命名

例如：

不要：

```text id="65a7kc"
tmp

data1

obj

list2

flag
```

---

# 第五阶段：职责检查

确认职责是否清晰。

页面：

是否承担过多业务？

组件：

是否承担页面职责？

API：

是否包含业务逻辑？

Store：

是否维护页面状态？

Utils：

是否依赖页面？

Composable：

是否承担 UI？

发现职责混乱，应立即调整。

---

# 第六阶段：复杂度检查

检查代码复杂度。

包括：

* 函数长度
* 文件长度
* 嵌套层级
* 条件复杂度
* 重复代码
* 大型组件
* 大型 Store

如果明显超出项目规范，应拆分。

---

# 第七阶段：异常处理检查

确认所有异常均已处理。

包括：

* Promise
* async / await
* API 请求
* 表单提交
* 上传
* 下载

禁止：

空 `catch`。

禁止：

忽略异常。

---

# 第八阶段：状态检查

检查状态来源。

包括：

* 是否重复维护数据
* 是否存在双向来源
* 是否误用 Store
* 是否误用响应式
* 是否存在不必要的 Watch

保持：

单一数据来源。

---

# 第九阶段：性能检查

确认是否存在明显性能问题。

包括：

* 重复请求
* 重复计算
* 重复渲染
* 无意义 Watch
* 大量深拷贝
* 大量创建对象

注意：

不要为了性能而牺牲可读性。

---

# 第十阶段：样式检查

确认样式符合项目规范。

包括：

* 是否使用 Scoped
* 是否使用 CSS Variables
* 是否硬编码颜色
* 是否大量使用 `!important`
* 是否大量使用 `:deep()`
* 是否影响其它页面

---

# 第十一阶段：兼容性检查

确认修改不会破坏已有功能。

包括：

* Props
* Emits
* API
* Store
* Router
* 返回值
* 方法签名

除非需求明确要求，否则应保持兼容。

---

# 第十二阶段：可维护性检查

确认未来维护成本。

包括：

* 是否容易阅读
* 是否容易定位
* 是否容易扩展
* 是否容易测试
* 是否容易删除

如果半年后再次阅读仍能快速理解，则认为通过。

---

# 第十三阶段：安全检查

确认不存在明显安全风险。

包括：

* 是否存在 XSS 风险
* 是否直接拼接 HTML
* 是否暴露敏感信息
* 是否输出 Token
* 是否输出用户隐私
* 是否信任未校验输入

遵循项目安全规范。

---

# 第十四阶段：AI 幻觉检查

AI 必须主动确认：

是否存在：

* 猜测 API
* 猜测字段
* 猜测目录
* 猜测组件
* 猜测 Store
* 猜测工具函数

如果项目未提供：

应明确说明。

不要自行创造。

---

# 第十五阶段：最终一致性检查

最后确认：

整个修改是否：

* 与项目风格一致
* 与已有代码一致
* 与团队规范一致
* 与 AFES 规范一致

保证：

新代码看起来像一直存在于项目中。

---

# AI 自检流程

AI 每次完成开发后，应按照以下顺序执行：

1. 检查需求是否完成
2. 检查架构是否正确
3. 检查是否存在重复实现
4. 检查命名是否统一
5. 检查职责是否清晰
6. 检查复杂度是否合理
7. 检查异常处理是否完整
8. 检查状态管理是否合理
9. 检查性能是否存在明显问题
10. 检查样式是否符合规范
11. 检查兼容性
12. 检查可维护性
13. 检查安全性
14. 检查是否存在 AI 幻觉
15. 最终检查整体一致性

完成全部检查后，方可认为任务完成。

---

# AI 自动修正要求

如果自检发现问题：

AI 应优先：

1. 修复问题
2. 再次执行完整自检
3. 确认问题已消失

不要：

带着已知问题结束任务。

---

# AI 禁止行为

AI 不得出现以下行为：

* 未完成需求即结束任务
* 跳过自检
* 明知存在问题仍输出结果
* 为了通过检查而修改业务需求
* 猜测不存在的接口或组件
* 修改已有公共能力导致兼容性问题
* 引入新的编码风格
* 忽略重复实现
* 忽略异常处理
* 忽略安全问题
* 忽略可维护性问题

---

# AI 输出要求

完成自检后，AI 应确保最终结果满足以下标准：

* 满足用户需求
* 符合项目架构
* 符合 AFES 全部规范
* 保持项目风格一致
* 不引入重复实现
* 不破坏已有能力
* 命名统一
* 职责清晰
* 易于阅读
* 易于维护
* 易于扩展
* 易于测试
* 易于团队协作

---

# 最终 Checklist

AI 完成任务后，应确认以下内容全部通过：

## 需求

* 已完整实现用户需求
* 未新增无关功能
* 未改变既有业务行为

## 架构

* 目录正确
* 模块划分合理
* 职责边界清晰
* 未新增重复模块

## 复用

* 已检查并复用已有组件
* 已检查并复用已有 API
* 已检查并复用已有 Store
* 已检查并复用已有工具函数
* 已检查并复用已有样式

## 命名

* 文件命名规范
* 方法命名规范
* 变量命名规范
* Props 与 Emits 命名规范
* 无临时命名

## 代码质量

* 函数职责单一
* 文件复杂度合理
* 无重复代码
* 无未使用代码
* 异常处理完整
* 无空 `catch`

## 状态管理

* 数据来源唯一
* 未滥用 Store
* 未重复维护状态
* 响应式使用合理

## 样式

* 样式作用域正确
* 未污染全局
* 未滥用 `!important`
* 未滥用 `:deep()`
* 优先复用项目样式体系

## 安全

* 无明显 XSS 风险
* 未暴露敏感信息
* 未信任未校验输入

## 一致性

* 与项目代码风格一致
* 与 AFES 规范一致
* 与团队开发规范一致
* 最终代码可长期维护
