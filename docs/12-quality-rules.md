# 12 代码质量规则（Code Quality Rules）

## 目标

本规范定义 AI 在开发、修改、重构和维护代码时必须遵循的质量标准。

`07-coding-rules` 约束的是**如何编写代码**。

本规范约束的是**如何保证代码质量、可维护性和长期演进能力**。

AI 不仅要保证代码能够运行，更要保证代码能够被团队长期维护。

一句话：

> 能运行只是最低要求，可维护才是最终目标。

---

# 核心原则

AI 必须始终遵循以下原则：

1. 正确性优先于速度
2. 可读性优先于技巧
3. 一致性优先于个性
4. 简单性优先于复杂性
5. 复用优先于重复
6. 稳定性优先于炫技
7. 最小修改优先于大规模重构

任何优化都不得以降低代码可维护性为代价。

---

# 正确性优先

AI 生成代码前，应确保：

* 满足需求
* 不破坏已有功能
* 不引入明显 Bug
* 不违反项目规范

不要：

为了优化代码：

改变业务行为。

---

# 一致性优先

整个项目：

应保持：

统一：

* 命名
* 风格
* 目录
* 注释
* 异常处理
* 导入顺序

不要：

同一项目：

出现：

三四种编码风格。

---

# 最小修改原则

修改已有代码：

优先：

局部修改。

例如：

修复：

一个判断。

不要：

整个页面：

全部重写。

保持：

Diff：

尽可能小。

---

# 可读性优先

代码：

应做到：

无需解释即可理解。

推荐：

```text id="0zcyt9"
const hasPermission = ...
```

不要：

```text id="q4ecq2"
const flag = ...
```

可读性优先于代码长度。

---

# 单一职责

一个：

模块。

一个：

组件。

一个：

函数。

一个：

Store。

都应：

只负责：

一类事情。

不要：

一个函数：

完成整个业务流程。

---

# 函数设计

建议：

一个函数：

只做：

一个动作。

例如：

```text id="gfquv7"
loadList()

validateForm()

buildTree()

submitForm()
```

不要：

```text id="ujjlwm"
handleEverything()
```

---

# 函数长度

建议：

控制：

30 行以内。

超过：

建议：

继续拆分。

复杂逻辑：

拆成：

多个：

小函数。

---

# 参数数量

建议：

函数参数：

不超过：

4 个。

超过：

建议：

使用：

对象参数。

例如：

```text id="z1wrvj"
options
```

提高：

可维护性。

---

# 返回值

保持：

统一。

一个函数：

尽量：

一种返回结构。

不要：

有时：

返回：

数组。

有时：

返回：

对象。

有时：

返回：

Boolean。

---

# 命名质量

命名必须体现：

业务含义。

不要：

```text id="gy4yzg"
data

info

obj

tmp

flag

aaa
```

推荐：

```text id="74cz1u"
userList

queryParams

selectedRows

hasPermission
```

---

# 魔法值

禁止：

大量：

硬编码。

例如：

```text id="i0w60o"
0

1

2

99

100

5000
```

建议：

提取：

常量。

保持：

业务语义。

---

# 条件判断

保持：

简单。

推荐：

提前返回。

例如：

```text id="sotpqb"
if (!user) return
```

不要：

大量：

```text id="fwg7vj"
if

↓

else

↓

else

↓

else
```

---

# 深层嵌套

建议：

最多：

三层。

超过：

应：

* 拆函数
* 提前返回
* 使用策略模式（项目已有时）
* 提取条件

避免：

阅读困难。

---

# 重复代码

出现：

三次以上：

建议：

抽离。

例如：

重复：

* 日期处理
* Message
* 表格格式化
* 权限判断

不要：

复制粘贴。

---

# 异常处理

所有：

异步逻辑：

必须：

处理异常。

不要：

空：

```text id="zfpb77"
catch {}
```

不要：

忽略：

Promise。

错误应：

可发现。

可定位。

---

# 日志

开发：

允许：

```text id="bw4pqd"
console.log()
```

提交前：

删除。

允许保留：

```text id="96y7pb"
console.warn()

console.error()
```

前提：

符合项目规范。

---

# 注释质量

注释：

解释：

为什么。

不是：

解释：

代码。

推荐：

* 特殊业务
* 第三方限制
* 历史兼容
* 性能原因

不要：

```text id="dtpv1d"
循环数组

获取数据

点击按钮
```

---

# TODO

允许：

TODO。

必须：

写明：

原因。

例如：

```text id="bdz96i"
TODO：等待后端新增批量审批接口后优化
```

不要：

只有：

```text id="cxm36l"
TODO
```

---

# Import

统一排序：

1. Vue
2. 第三方
3. API
4. Store
5. Hooks
6. Utils
7. Components
8. Styles

不要：

随机顺序。

---

# 未使用代码

禁止：

保留：

* 未使用 Import
* 未使用变量
* 未使用方法
* 未使用类型
* 未使用常量

保持：

代码整洁。

---

# 副作用

函数：

尽量：

保持纯函数。

避免：

隐藏副作用。

例如：

不要：

函数：

偷偷：

修改：

全局变量。

---

# 工具函数

Utils：

应：

独立。

不要：

依赖：

页面。

不要：

依赖：

Store。

不要：

依赖：

DOM。

---

# Composable

Composable：

负责：

逻辑复用。

不要：

负责：

页面布局。

不要：

负责：

UI。

不要：

承担多个业务。

---

# Store

Store：

负责：

全局状态。

不要：

存放：

页面状态。

不要：

承担：

页面业务。

---

# 页面

页面：

负责：

组织业务。

不要：

承担：

公共工具。

不要：

承担：

复杂算法。

---

# API

API：

负责：

请求。

不要：

负责：

业务。

不要：

负责：

UI。

---

# 组件

组件：

负责：

UI。

不要：

负责：

页面。

不要：

负责：

全局状态。

---

# 重构原则

允许：

重构。

必须：

满足：

* 不改变行为
* 不改变接口
* 不增加复杂度
* 提高可读性

不要：

为了：

"更优雅"

重写整个模块。

---

# 向后兼容

修改已有能力：

优先：

保持兼容。

例如：

不要：

修改：

已有：

* Props
* Emits
* API
* Store
* 方法名称
* 返回值

除非：

需求明确要求。

---

# AI 自检流程

AI 在完成代码后，应至少完成以下检查：

1. 是否复用已有能力
2. 是否符合目录规范
3. 是否符合命名规范
4. 是否存在重复代码
5. 是否存在无用代码
6. 是否存在异常遗漏
7. 是否影响已有功能
8. 是否符合项目整体风格
9. 是否容易维护
10. 是否容易扩展

任何一项未通过，都应继续优化。

---

# AI 自动生成要求

AI 在生成代码时必须遵守：

* 优先复用已有实现
* 保持最小修改原则
* 保持项目风格一致
* 使用语义化命名
* 保持职责单一
* 保持函数简洁
* 控制复杂度
* 保持异常处理完整
* 保持代码整洁
* 保持向后兼容
* 保持长期可维护性

---

# AI 禁止行为

AI 不得生成以下代码：

* 大规模无必要重构
* 重复实现已有能力
* 深层嵌套逻辑
* 超长函数
* 超级组件
* 超级 Store
* 魔法值遍布代码
* 大量硬编码字符串
* 空 `catch`
* 大量调试日志
* 无意义注释
* 未使用代码
* 无语义命名
* 引入新的框架或基础设施
* 修改已有公共接口导致兼容性问题
* 与项目整体开发规范不一致

---

# Checklist

AI 完成开发后，应确认以下内容：

* 已优先复用项目已有实现
* 修改范围符合最小修改原则
* 保持与项目整体编码风格一致
* 命名清晰且具有业务语义
* 函数职责单一，长度合理
* 条件判断简单，避免深层嵌套
* 魔法值已提取为常量（适用时）
* 重复代码已抽离
* 异常处理完整，无空 `catch`
* 无未使用的导入、变量或方法
* 注释具有实际价值
* 工具函数保持纯函数特性
* 页面、组件、Store、API 职责边界清晰
* 保持向后兼容，不破坏已有能力
* 新增代码易于阅读、测试、维护与扩展
* 满足项目长期演进与团队协作要求
