# 15 AI 禁止规则（Forbidden Rules）

## 目标

本规范定义 AFES（AI Frontend Engineering Specification）中 AI 必须遵守的禁止性规则。

前面的规范主要告诉 AI **应该做什么**。

本规范明确规定 AI **绝不能做什么**。

任何禁止规则的优先级高于建议性规则。

一句话：

> 不允许出现的行为，比推荐的实现方式更加重要。

---

# 总原则

AI 在任何情况下都不得：

* 猜测项目结构
* 猜测业务逻辑
* 猜测接口定义
* 猜测数据库字段
* 猜测组件能力
* 猜测 Store 能力
* 猜测工具函数
* 猜测第三方依赖

如果缺少必要信息：

应明确说明，并请求补充。

不得自行创造不存在的内容。

---

# 禁止虚构项目能力

AI 不得假设项目已经存在：

* API
* Store
* Hooks（Composable）
* Utils
* Components
* Directive
* Plugin
* Types
* Enum
* Constant
* Mock

例如：

项目未提供：

```text
src/api/user.js
```

不得直接调用：

```text
listUser()
```

除非用户已经提供相关实现。

---

# 禁止修改需求

AI 不得：

* 擅自扩大需求范围
* 擅自缩小需求范围
* 擅自改变业务流程
* 擅自改变交互方式
* 擅自改变数据结构

例如：

用户要求：

新增一个查询条件。

不得：

顺便重构整个查询模块。

---

# 禁止大规模重构

局部问题：

只能局部解决。

不得：

为了修改：

一个按钮。

重写整个页面。

不得：

为了修改：

一个组件。

重写整个模块。

保持：

最小修改原则。

---

# 禁止重复实现

AI 不得：

重复创建：

* API
* Store
* Component
* Hook
* Utils
* 样式
* 常量
* 枚举

新增之前：

必须：

优先检查是否已有实现。

---

# 禁止破坏兼容性

不得：

修改已有：

* API 参数
* API 返回值
* Props
* Emits
* Store State
* Store Action
* 方法签名
* 公共组件行为

除非：

用户明确要求。

---

# 禁止职责混乱

页面：

不得承担：

* 工具函数
* HTTP 封装
* 全局状态

组件：

不得承担：

* 页面管理
* Router
* 全局业务流程

API：

不得承担：

* Message
* Loading
* Dialog
* Router
* Store

Store：

不得承担：

* 页面状态
* UI 状态
* 页面生命周期

Utils：

不得依赖：

* 页面
* Store
* Router

---

# 禁止绕过项目规范

AI 不得：

为了实现需求：

绕过：

* 权限体系
* 路由体系
* 请求封装
* 状态管理
* 组件规范
* 样式规范

例如：

不得：

页面直接：

```text
fetch(...)
```

如果项目统一使用：

Request 封装。

---

# 禁止引入新技术方案

除非用户明确要求。

否则：

不得：

新增：

* 新框架
* 新状态管理
* 新 UI 库
* 新请求库
* 新构建工具
* 新 CSS 方案
* 新目录规范

保持：

项目已有技术栈。

---

# 禁止隐藏风险

如果修改存在风险。

AI 必须：

主动说明。

例如：

* 影响多个模块
* 修改公共组件
* 修改公共 API
* 修改公共 Store
* 修改数据库字段（如涉及）

不得：

隐瞒。

---

# 禁止忽略异常处理

AI 不得：

新增：

没有异常处理：

的：

* API 请求
* 表单提交
* 上传
* 下载
* 异步操作

不得：

出现：

空：

```text
catch {}
```

---

# 禁止保留调试代码

提交代码时：

不得：

保留：

* `console.log`
* `debugger`
* 测试按钮
* 临时变量
* 临时代码
* 调试注释

除非：

用户明确要求。

---

# 禁止无意义代码

不得：

新增：

* 未使用变量
* 未使用 Import
* 未使用函数
* 未使用类型
* 未使用常量

不得：

保留：

废弃代码。

---

# 禁止无意义抽象

不要：

为了：

"以后可能用到"

新增：

* BaseXXX
* AbstractXXX
* Manager
* Factory
* Builder

除非：

项目已经采用相同设计。

遵循：

YAGNI（You Aren't Gonna Need It）原则。

---

# 禁止过度设计

不要：

一个简单页面：

拆分：

二十多个组件。

不要：

一个简单函数：

设计：

多层抽象。

优先：

简单。

---

# 禁止深层嵌套

避免：

超过：

三层：

嵌套。

例如：

```text
if
 └── for
      └── if
           └── while
```

应：

拆分。

提前返回。

---

# 禁止魔法值

不得：

大量：

出现：

```text
0
1
2
99
100
999
```

应：

提取：

常量。

保持：

业务语义。

---

# 禁止硬编码

不得：

硬编码：

* URL
* Token
* 用户 ID
* 角色 ID
* 权限编码
* 接口地址
* 颜色
* 文案（可配置内容）

应：

统一配置。

---

# 禁止污染全局

不得：

新增：

* 全局变量
* Window 挂载
* Prototype 扩展
* 全局样式污染
* 全局事件

除非：

项目已有规范。

---

# 禁止样式污染

不得：

页面：

修改：

其它页面样式。

不得：

组件：

修改：

其它组件样式。

默认：

使用：

`scoped`。

---

# 禁止修改第三方源码

不得：

直接修改：

* Element Plus
* Vue
* Pinia
* Vite
* 第三方插件

应：

通过配置或扩展实现。

---

# 禁止忽略性能问题

不得：

明显：

重复：

* 请求接口
* 创建对象
* 深拷贝
* Watch
* 大量计算

但：

也不得：

为了性能：

牺牲：

可读性。

---

# 禁止忽略安全问题

不得：

直接：

渲染：

未校验 HTML。

不得：

暴露：

Token。

不得：

输出：

敏感信息。

不得：

信任：

用户输入。

遵循：

项目安全规范。

---

# 禁止 AI 幻觉

AI 不得：

将：

推测：

描述为：

事实。

例如：

不得：

> 项目中已经有 useUser Hook。

如果用户没有提供。

应：

说明：

无法确认。

---

# AI 强制中止条件

遇到以下情况：

AI 应停止生成代码，并说明原因：

* 缺少关键需求
* 缺少目录结构
* 缺少接口定义
* 缺少已有代码
* 存在多个合理方案且无法判断
* 修改可能导致重大兼容性风险

不得：

继续猜测实现。

---

# AI 强制遵循原则

任何时候：

都必须遵循：

1. 用户需求优先
2. 项目规范优先
3. 最小修改原则
4. 复用优先
5. 一致性优先
6. 可维护优先
7. 可验证优先

以上原则：

优先于个人偏好。

---

# AI 禁止行为清单

AI 不得：

* 猜测项目能力
* 虚构 API、组件、Store 或工具函数
* 修改用户需求
* 无必要大规模重构
* 重复实现已有能力
* 破坏向后兼容
* 职责混乱
* 绕过项目架构
* 引入新的技术方案
* 隐瞒风险
* 忽略异常处理
* 保留调试代码
* 保留无用代码
* 进行过度抽象或过度设计
* 使用深层嵌套
* 大量使用魔法值
* 硬编码关键配置
* 污染全局环境
* 污染其他页面或组件样式
* 修改第三方源码
* 忽略明显性能问题
* 忽略安全风险
* 将推测描述为事实
* 在缺少关键信息时继续猜测生成代码

---

# 最终 Checklist

AI 输出前，应确认以下内容全部通过：

## 需求

* 未修改用户需求
* 未扩大实现范围
* 未缩小实现范围

## 项目

* 未猜测项目结构
* 未虚构已有能力
* 已优先考虑复用

## 架构

* 未破坏职责边界
* 未绕过项目规范
* 未引入新的技术方案

## 代码

* 未进行无必要重构
* 未新增重复实现
* 未保留调试代码
* 未保留无用代码
* 未引入兼容性问题

## 样式

* 未污染全局样式
* 未污染其他组件
* 未大量使用硬编码样式

## 安全

* 未暴露敏感信息
* 未信任未校验输入
* 未引入明显安全风险

## 输出

* 未将推测描述为事实
* 已说明潜在风险
* 缺少关键信息时已停止猜测
* 输出符合 AFES 全部规范
* 输出结果可长期维护、可直接用于团队协作
